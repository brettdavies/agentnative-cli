# Spike: Go, Ruby, and TypeScript source-check viability

## Purpose

Unit 1 of the multi-language source-checks plan. The plan could not commit pattern shapes, dispatch layout, or a
grammar-cost budget without empirical validation, so this spike probes `ast-grep-language 0.42.0` against representative
Go, Ruby, and TypeScript source to confirm:

- The exact feature-flag and type exports for each new grammar.
- The meta-variable syntax users must write.
- The tree-sitter node-kind names our AST-walking checks will target.
- The stripped-release binary-size delta per grammar.
- The clean-build compile-time cost per grammar.
- Any pattern-shape surprises that change the implementation strategy.

Units 2–5 of the plan depend on this report. Findings below are empirical — every match/no-match claim comes from a
scratch crate built against the pinned `=0.42.0` dependencies, not from documentation.

## TL;DR

- All three grammars are first-class in `ast-grep-language 0.42.0` and ready to enable. Feature flags: `tree-sitter-go`,
  `tree-sitter-ruby`, `tree-sitter-typescript` (no separate `tree-sitter-tsx` feature; `Tsx` ships inside the TypeScript
  crate).
- **The meta-var risk the plan flagged is defused.** `Rust` and `Python` are *also* registered via `impl_lang_expando!`
  with expando char `µ`. The library rewrites `$VAR` → `µVAR` transparently at pattern-parse time for every
  `impl_lang_expando!` language. Users still write `$VAR` for Go, Ruby, TypeScript, Python, Rust — same as today. No
  cross-language helper change is needed to "handle" `µ`.
- **Go has a real pattern-parse gotcha.** Top-level bare expressions like `os.Exit($CODE)` fail to parse as a call —
  tree-sitter-go returns an `ERROR` node wrapping a `type_conversion_expression`. Patterns targeting selector calls must
  be wrapped (e.g. `func $F() { os.Exit($CODE) }`) or the check must walk the AST directly. Bare function calls
  (`panic($MSG)`) parse fine. This is the single most important planning adjustment.
- Binary-size deltas (stripped release, matching agentnative's release profile): **Go ≈ +220 KB, Ruby ≈ +2.1 MB,
  TypeScript ≈ +2.9 MB, all three ≈ +5.2 MB.** Well under the plan's 15 MB total cap and the 5 MB per-grammar cap.
- Clean-build time in the spike crate is 9s with and without the three extra grammars — the tree-sitter parsers
  themselves are cheap to compile. The main cost is transferred to binary size, not build time.
- The Gemfile / `*.gemspec` / `tsconfig.json` detection rules assumed by the plan remain correct; no manifest-ordering
  surprise emerged.
- **Recommendation:** proceed to Unit 2 as planned. Tighten Unit 3 (Go) to prefer AST walking over bare expression
  patterns; keep Units 4 (TypeScript) and 5 (Ruby) as drafted. Two open questions surfaced below.

## Methodology

A scratch crate at `/tmp/anc-spike/` was created with the same `[profile.release]` as `agentnative` (`strip = true`,
`lto = true`, `codegen-units = 1`, `panic = "abort"`), pinning `ast-grep-core = 0.42.0` and `ast-grep-language =
0.42.0`. Four probe binaries were built:

1. **Baseline** — `tree-sitter-rust` + `tree-sitter-python` only.
2. **+Go** — baseline plus `tree-sitter-go`.
3. **+Ruby** — baseline plus `tree-sitter-ruby`.
4. **+TypeScript** — baseline plus `tree-sitter-typescript` (which pulls in both `TypeScript` and `Tsx`).
5. **+all three** — baseline plus all three features.

Pattern probes used `ast_grep_core::Pattern::try_new(pat, lang)` followed by `lang.ast_grep(src).root().find_all(&p)`.
AST dumps were generated by walking `Node::children()` recursively. Node-kind lists were gathered via the same walk and
deduplicated.

No source code under `src/` was modified. No Cargo.toml change landed in the repository. The scratch crate is disposable
and lives outside the repo root.

## Per-grammar findings

### Go

| Field | Value |
| ----- | ----- |
| Cargo feature flag | `tree-sitter-go` |
| Exposed Rust type | `ast_grep_language::Go` |
| Registration macro | `impl_lang_expando!(Go, language_go, 'µ')` |
| Grammar version | `tree-sitter-go = 0.25.0` |
| User-facing meta-var | `$VAR` (rewritten to `µVAR` internally) |
| Binary-size delta | +225,200 bytes (~220 KB) |
| Clean-build-time delta | negligible (within ±1s of baseline 9s) |

**Pattern-parse behavior.** The top-level Go pattern source must either be a complete Go declaration (function, import,
package) or a bare expression the Go grammar accepts as a top-level snippet. `panic("boom")` parses cleanly as a
`call_expression` at the top level. `os.Exit(0)`, `os.Getenv("NO_COLOR")`, and `log.Fatalf(...)` — all
selector-expression calls — fail: tree-sitter-go treats the bare `a.b` portion as a `qualified_type`, interprets `(arg)`
as a type conversion, and wraps the whole thing in an `ERROR` node.

Workarounds tested and working:

- **Wrap in a function**: `func $F() { os.Exit($CODE) }` parses fine and matches any `os.Exit($CODE)` inside any
  function body.
- **Provide real expected args**: `log.Fatalf($FMT, $$$REST)` matches `log.Fatalf("%v", err)`, because the two-argument
  shape forces the parser to choose `call_expression` over `type_conversion_expression`.
- **AST walking**: walk `call_expression` nodes, inspect the first child (selector_expression with package_identifier
  and field_identifier children). This is the most robust option and mirrors the Python `bare_except` / `sys_exit` idiom
  already in the codebase.

**Recommended strategy for Go checks.** Use AST walking for any check whose target is a selector-expression call
(`os.Exit`, `log.Fatal*`, `os.Getenv`, etc.). Reserve `Pattern` for unqualified builtins (`panic`, `print`, `println`).

**Sample node kinds observed** (from a representative `package main` file):

- `source_file`, `package_clause`, `package_identifier`
- `import_declaration`, `import_spec_list`, `import_spec`
- `function_declaration`, `parameter_list`, `block`, `statement_list`
- `short_var_declaration`, `expression_statement`, `expression_list`
- `call_expression`, `argument_list`, `selector_expression`, `field_identifier`
- `if_statement`, `binary_expression`, `return_statement`, `interpreted_string_literal`

**Starter check candidates (4, final list for Unit 3):**

| ID | Principle | Template | Strategy | Notes |
| -- | --------- | -------- | -------- | ----- |
| `code-go-panic` | code-quality | `src/checks/source/rust/unwrap.rs` | `Pattern::try_new("panic($MSG)", Go)` — works at top level. Exempt `main.go` + explicit `// nolint:agentnative-panic` comments (future). | Spike confirmed `panic($MSG)` matches. |
| `p4-go-os-exit` | P4 | `src/checks/source/python/sys_exit.rs` | AST walk — `call_expression > selector_expression(package_identifier="os", field_identifier="Exit")`. Skip files whose `package_clause` is `package main` (or whose path basename is `main.go`). | Pattern-form fails at top level; AST walk is required. |
| `p4-go-log-fatal` | P4 | `src/checks/source/python/sys_exit.rs` | AST walk — `call_expression > selector_expression(package_identifier="log", field_identifier ∈ {"Fatal","Fatalf","Fatalln"})`. Same main-package exemption. | New idiom; direct Go analog to Python's `sys_exit`. Log.Fatal family terminates the process. |
| `p6-go-no-color` | P6 | `src/checks/source/python/no_color.rs` | Pattern `os.Getenv($KEY)` plus AST walk for selector shape, plus `has_string_literal_in(source, "NO_COLOR", Go)` fallback. | String-literal fallback is needed because patterns with the bare form fail at top level. |

### Ruby

| Field | Value |
| ----- | ----- |
| Cargo feature flag | `tree-sitter-ruby` |
| Exposed Rust type | `ast_grep_language::Ruby` |
| Registration macro | `impl_lang_expando!(Ruby, language_ruby, 'µ')` |
| Grammar version | `tree-sitter-ruby = 0.23.0` (resolved to 0.23.1 in lockfile — upstream patch release) |
| User-facing meta-var | `$VAR` (rewritten to `µVAR` internally). **Caveat: Ruby global variables named with uppercase letters, e.g. `$PROGRAM_NAME`, are also rewritten by the same rule — writing the literal token in a pattern makes it a meta-var, not a match. `$0` is safe (digit, not rewritten).** |
| Binary-size delta | +2,102,704 bytes (~2.1 MB) |
| Clean-build-time delta | negligible |

**Pattern-parse behavior.** All probed expression patterns work: `exit $CODE`, `exit($CODE)`, `raise $MSG`,
`ENV["NO_COLOR"]`, `ENV[$K]`. The earlier plan assumption that parens matter (`exit($CODE)` vs `exit $CODE`) was
incorrect — the spike confirms both forms match the same Ruby `call` node with `argument_list` children.

**Rescue-node shape** (for the `code-ruby-bare-rescue` check):

- Bare rescue: `rescue [rescue => e]` with child structure `rescue_keyword + exception_variable + then + body`. No
  `exceptions` child node.
- Typed rescue: `rescue [rescue StandardError => e]` with child structure `rescue_keyword + exceptions +
  exception_variable + then + body`. An `exceptions` child is present.

This makes the bare-rescue check a direct analog of Python's `bare_except`: walk for `rescue` nodes and check whether
any child has kind `"exceptions"`.

**Entry-point idiom detection.** Patterns containing Ruby's uppercase global variable `$PROGRAM_NAME` cannot be matched
literally via `Pattern::try_new` (the uppercase triggers meta-var rewriting). The canonical `if __FILE__ == $0` form
*does* match literally because `$0` is a digit global. For uniform support of both forms, use AST walking and inspect
the `if_statement` header text — exactly what Python's `sys_exit::is_main_guard` does with `__name__ == "__main__"`.

**Sample node kinds observed:**

- `program`, `method`, `body_statement`, `begin`, `if`, `binary`
- `call`, `identifier`, `argument_list`, `element_reference`
- `string`, `string_content`, `interpolation`, `integer`, `constant`
- `rescue`, `exception_variable`, `exceptions`, `then`, `end`
- `global_variable` (e.g., `$0`, `$PROGRAM_NAME`)

**Starter check candidates (4, final list for Unit 5):**

| ID | Principle | Template | Strategy | Notes |
| -- | --------- | -------- | -------- | ----- |
| `code-ruby-bare-rescue` | code-quality | `src/checks/source/python/bare_except.rs` | AST walk for `rescue` nodes; fail when no child has kind `"exceptions"`. | Direct structural analog of Python's bare-except. |
| `p4-ruby-exit` | P4 | `src/checks/source/python/sys_exit.rs` | AST walk for `call` nodes whose `identifier` child text is `"exit"`, `"exit!"`, or `"abort"`; allow inside `if __FILE__ == $0` or `if $PROGRAM_NAME == __FILE__` guards (detected via `if_statement` header-text inspection, not pattern). | Mirrors Python's guard-aware `sys_exit` check. |
| `p6-ruby-no-color` | P6 | `src/checks/source/python/no_color.rs` | Patterns: `ENV[$K]`, `ENV.fetch($K)`, `ENV.fetch($K, $DEFAULT)`. Plus `has_string_literal_in(source, "NO_COLOR", Ruby)` fallback. | Patterns work directly (confirmed in spike). |
| `p4-ruby-raise-string` | P4 | partial analog of `src/checks/source/rust/error_types.rs` | Pattern `raise $MSG` where `$MSG` resolves to a `string` node (not a `constant` or call). Requires small helper to inspect `$MSG`'s kind — or walk AST for `call` where `identifier.text == "raise"` and first argument kind is `string`. | Encourages StandardError subclasses; direct analog to the "typed errors" family. |

### TypeScript

| Field | Value |
| ----- | ----- |
| Cargo feature flag | `tree-sitter-typescript` (ships `TypeScript` and `Tsx` in one crate) |
| Exposed Rust types | `ast_grep_language::TypeScript`, `ast_grep_language::Tsx` |
| Registration macro | `impl_lang!(TypeScript, language_typescript)` and `impl_lang!(Tsx, language_tsx)` — **no expando, `$` works natively** |
| Grammar version | `tree-sitter-typescript = 0.23.2` |
| User-facing meta-var | `$VAR` (no rewriting; `$` is not a valid identifier char in TS) |
| Binary-size delta | +2,880,864 bytes (~2.9 MB) |
| Clean-build-time delta | negligible |

**Pattern-parse behavior.** Every expression pattern probed matched cleanly: `process.exit($CODE)`,
`console.log($$$ARGS)`, `process.env.NO_COLOR`, `process.env[$K]`, `throw new Error($MSG)`. No wrapping needed. TSX also
works — `process.exit($CODE)` matches inside a JSX expression container.

**TypeScript vs Tsx handling.** For the starter check set, parsing every `.ts` / `.tsx` file with the `TypeScript`
grammar is likely sufficient (no JSX-specific checks yet). If a future check needs JSX-awareness, dispatch `.tsx` files
to `Tsx` and `.ts` files to `TypeScript`. Both grammars ship in the same feature, so there's no additional cost to
keeping the choice open.

- `new_expression`, `lexical_declaration`, `variable_declarator`
- `property_identifier`, `identifier`, `string`, `string_fragment`, `number`
- `type_annotation`, `predefined_type`, `unary_expression`
- Tsx-specific: `jsx_element`, `jsx_opening_element`, `jsx_closing_element`, `jsx_attribute`, `jsx_expression`,

## Cross-language observations

- **`impl_lang_expando!` is shared infrastructure, not a distinguishing hazard.** `Rust`, `Python`, `Go`, `Ruby`, `C`,
  `Cpp`, `CSharp`, `Css`, `Elixir`, `Haskell`, `Hcl`, `Kotlin`, `Nix`, `Php`, `Swift` all use it. `Bash`, `Java`,
  `JavaScript`, `Json`, `Lua`, `Scala`, `Solidity`, `Tsx`, `TypeScript`, `Yaml` use the simpler `impl_lang!`. The
  distinction only affects internals; user-facing pattern syntax is `$VAR` for all supported languages.
- **Meta-var rewriting has edge cases in Ruby.** Uppercase-only globals (`$PROGRAM_NAME`) cannot be matched as literals
  via `Pattern`. AST walking is the fallback. `$0` through `$9` are digit globals and pass through unchanged.
- **Top-level pattern parsing is language-dependent.** Go is strict (snippets must be valid top-level declarations or
  simple builtin calls). Python, Ruby, TypeScript are all permissive (expressions parse at top level). This is the root
  cause of the Go check-strategy asymmetry.
- **Pattern::try_new is cheap enough that the "pre-build patterns" convention from `CLAUDE.md` remains the right
  default.** Patterns used repeatedly in a check (one per file in `run()`) should still be lazily built once per check
  invocation, not per call site.

## Manifest detection

The plan's proposed ordering survives spike review:

```text
Cargo.toml        -> Rust
pyproject.toml    -> Python
tsconfig.json     -> TypeScript   (inserted before package.json)
package.json      -> Node         (unchanged; JS fallback)
go.mod            -> Go
Gemfile           -> Ruby
<any>.gemspec     -> Ruby         (directory scan for *.gemspec suffix)
```

Two small implementation notes for Unit 2:

- `*.gemspec` detection needs a `read_dir` + extension check — not a direct `file.exists()` call. Keep the cost low by
  short-circuiting after the first match.
- `tsconfig.json` is a plain JSON file the detector never needs to parse; `file.exists()` is sufficient. If a repo ships
  both `tsconfig.json` and `pyproject.toml` (rare: a Python project with bundled TS tooling), the current
  first-match-wins semantics classify it as Python — matching today's behavior for Python + Cargo.toml polyglot repos.

- Per-grammar cap: 5 MB stripped. Max observed: TypeScript at ~2.9 MB. ✅
- Total cap: 15 MB stripped. Observed total: ~5.2 MB. ✅

### Build time

Clean release build of the scratch crate, `lto = true` + `codegen-units = 1`, on this machine:

- Baseline: 9s
- With all three grammars: 9s

Tree-sitter grammars are mostly pre-compiled C with small Rust wrappers — they add link work but almost no compile
work. Expect agentnative's CI clean-build time to grow by ~5–10s total, likely dominated by LTO.

## Surprises

1. **Meta-var "risk" turned out to be a non-issue.** The plan flagged the `µ` expando as a potential cross-language
   helper hazard. Spike shows `Rust` and `Python` are on the same macro today — the "risk" was imaginary. Cross-language
   helpers can extend by analogy with zero special-casing.
2. **Go bare-expression patterns don't parse.** Genuine surprise and the main adjustment this spike makes to the plan.
   AST walking becomes the default check strategy for Go selector-expression targets.
3. **Ruby's global-variable meta-var collision.** `$PROGRAM_NAME` cannot be matched as a literal via `Pattern`. Ruby's
   entry-point detection must use AST walking + text inspection (same shape as Python's `is_main_guard`). Forewarned
   avoids a half-day wasted on "why doesn't my pattern match".
4. **`tree-sitter-typescript = 0.23.2` is a *much* larger binary contributor than Go.** Expected TS to be heavier than
   Go given the grammar's size, but 2.9 MB vs 220 KB is a 13× gap. If binary size ever becomes a release-artifact
   blocker (Homebrew formula, CI asset limits), TypeScript is the first grammar to consider feature-gating.
5. **Tsx ships inside `tree-sitter-typescript`, with no separate feature flag.** The plan's table for Unit 2 already
   reflects this, but it is worth restating: there is no need for a separate feature toggle for Tsx.

## Open questions for Unit 2+

1. **Should Go use AST walking uniformly, or a mix of patterns (for bare builtins) and AST walking (for selector
   calls)?** A uniform walk is simpler to reason about; a mix keeps `panic($MSG)` as a one-liner pattern. Recommend
   **mix**: pattern-based for bare builtin calls (`panic`, `println`, `print`), AST-walk for selector calls (`os.*`,
   `log.*`, `fmt.*`). Revisit at Unit 3 if maintenance cost of two shapes becomes visible.
2. **Should the Ruby `p4-ruby-raise-string` check flag `raise "msg"` as well as `raise RuntimeError, "msg"`?** The
   latter is idiomatic Ruby; flagging it would be noisy. Recommend flag only the bare form; document the exemption in
   the check's module doc comment.

## Recommendations for Unit 2

No changes to the plan's cross-cutting infrastructure. Specifically:

- Keep the `Language::TypeScript` and `Language::Ruby` variants proposed in the plan.
- Keep the multi-extension walk change (`TypeScript -> ["ts", "tsx"]`, `Ruby -> ["rb"]`).
- Keep the `src/source.rs` dispatch extension — no special handling needed for `µ` expando-char languages since
  `Rust`/`Python` already go through the same path today.
- Remove the "Ruby or Go `µ` meta-var breaks existing `$`-pattern assumptions in shared helpers" risk from the plan's
  Risks table — spike shows it is not a real risk.
- Add a new risk: **Go top-level pattern-parse failure for selector-expression calls**. Mitigation: Unit 3 uses AST
  walking for selector-call checks; patterns reserved for bare builtins. Sample Go checks ship with explicit tests that
  exercise both the pattern and the walker.

## Recommendations for Units 3–5

- **Unit 3 (Go)**: adopt the hybrid "pattern for builtins, AST walk for selector calls" strategy. Starter check list:
  `code-go-panic`, `p4-go-os-exit`, `p4-go-log-fatal`, `p6-go-no-color`.
- **Unit 4 (TypeScript)**: patterns work as-drafted. Starter check list: `p4-ts-process-exit`, `code-ts-console-log`,
  `p6-ts-no-color`, `p4-ts-throw-string`. Parse all `.ts` and `.tsx` files as `TypeScript` for v0.2; revisit Tsx split
  if a JSX-aware check lands.
- **Unit 5 (Ruby)**: rescue check uses AST walk (node kind `rescue`, detect absence of `exceptions` child). Exit check
  uses AST walk + header-text guard detection. `$PROGRAM_NAME` cannot appear literally in a pattern — write an AST
  walker. Starter check list: `code-ruby-bare-rescue`, `p4-ruby-exit`, `p6-ruby-no-color`, `p4-ruby-raise-string`.
